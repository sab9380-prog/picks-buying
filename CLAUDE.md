## 2. 행동 원칙 (사용자 선호)

### 2.1 추측 금지, 할루시네이션 매우 싫음
모르는 건 "모름"이라고 말한다. 코드를 안 보고 추측해서 답변하지 않는다. 파일 수정 전엔 반드시 Read로 본 후 작업한다.

### 2.2 무조건 동조 금지
사용자 의견에 무조건 따르지 말고, 비판적·객관적 더 나은 제안이 있으면 짚어둔 뒤 진행한다. 단, 사용자가 결정한 방향은 존중한다.

### 2.3 품질 높은 답변을 위한 조건값 묻기
요청이 모호하거나 결정 분기가 있으면 작업 시작 전에 묻는다. 작업 도중 모호한 부분이 또 나오면 다시 묻는다.

### 2.4 답변 형식
- 일상 대화는 짧고 자연스럽게
- 보고서·계획서는 마크다운 구조 사용 가능
- 불릿 남발 금지 (1~2문장은 그냥 문장으로)
- 한국어로 답한다

## 3. 코드 작업 룰

### 3.1 토큰 단일 출처
모든 색·폰트·간격·반경·그림자는 `design-tokens.css`(또는 프로젝트의 토큰 파일)에서만 정의한다. **JS·HTML·기타 CSS에 inline 색상 / 폰트 패밀리 / 픽셀 값 금지**. 새 값이 필요하면 토큰을 추가한다.

### 3.2 의존성 0 (smart-buy 한정)
smart-buy는 런타임 JS 라이브러리 추가를 0으로 유지한다. 예외는 PROJECT.md 3.1에 명시된 것뿐 (Pretendard, Inter, SheetJS).

### 3.3 폰트
- 한글: `var(--font-sans)` (Pretendard)
- 영문 본문: 부모 상속 또는 `var(--font-sans)` 폴백
- 영문 디스플레이·숫자: `var(--font-display)` (Inter)
- 코드·고정폭: `var(--font-mono)`
- 표·KPI 숫자에는 `font-variant-numeric: tabular-nums` 필수 (또는 `.num`/`.tabular` 클래스)

### 3.4 시스템 폰트 폴백 회피
모든 영역(표·입력폼·차트·디테일 패널 포함)에 토큰 명시. `font-family: inherit`은 부모가 토큰일 때만 OK.

### 3.5 mock vs 실 데이터
- 자동화는 "기능 동작" 검증만 가능. "MD가 쓸 만한가"는 mock으로 검증 안 됨.
- 휴리스틱·가정값 변경 작업은 실 BI 데이터 없이는 보류.
- mock 30 SKU 위에서의 결과로 판단하지 않는다.

### 3.6 검증 단계
변경 후 단위 테스트(`npm test:db / test:engine / test:normalize / test:oracle`) 모두 통과 확인. E2E는 사용자 머신에서 수행.

