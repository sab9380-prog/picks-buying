# DESIGN.md — Picks 디자인 시스템

> Picks 사업부 사내 도구의 디자인 가이드.
> 디자이너·개발자·기획자 누구나 이 문서만 보면 "이 도구는 이렇게 생겨야 한다"를 이해할 수 있어야 한다.
>
> Claude Code 행동 지침은 별도 문서(`CLAUDE.md`)에 있다.
> 이 문서는 *디자인의 무엇과 왜*에 집중한다.

---

## 1. 디자인 정체성

**Picks 도구는 B2C 친근함이 아니라 B2B 효율의 도구다.**

| 항목 | 정의 |
|---|---|
| **참조 모델** | Linear · Stripe Dashboard · Retool · Notion (이 순서) |
| **톤** | 차분하고 정돈된, 정보 밀도 높은 |
| **금기 톤** | 화려한, 친근한 마케팅, 게임 같은 |
| **타겟 환경** | 데스크톱 전용 (`min-width: 1280px`) |
| **다크 모드** | 옵션 아니라 기본 지원 (1시간 사용 → 눈 피로 대응) |

**한 줄 원칙**: "예쁘게"보다 "빠르게 결정 내릴 수 있게".

토스 스타일은 *라이팅(말투)에만* 적용한다 — 에러·빈 상태·확인 메시지의 한국어 톤. 시각 디자인은 토스가 아니라 Linear 계열이다.

---

## 2. 사용자 모델

디자인 결정이 흔들릴 때 **헤비유저를 우선**한다.

| 유형 | 누구 | 빈도 | 이 사용자에게 맞춰 디자인할 것 |
|---|---|---|---|
| **헤비유저** (1순위) | MD, 매입 담당 | 매일 1시간+ | 정보 밀도, 효율 |
| **라이트유저** (2순위) | 분석·기획팀 | 주 1~2회 | 처음 봐도 이해 가능 |
| **스파이크유저** (3순위) | 운영·영업 | 월 단위 | 길찾기 직관성 |

**불변 원칙**:
- 마우스만으로 모든 작업이 100% 가능해야 함
- 단축키는 *권장* 옵션, *필수* 아님
- 처음 본 사람도 5~10분 안에 핵심 작업 가능

## 3. 디자인 철학 (위에서부터 우선)

### 3.1 정보 밀도 우선
한 화면에 더 많이 보여라. 스크롤·탭 전환을 최소화한다. 여백을 위해 정보를 자르지 않는다. 사용자는 매일 보는 전문가다.

### 3.2 한 가지 정답 (Opinionated)
옵션·설정·토글을 추가하기 전에 항상 묻는다: "이거 정말 필요한가?" 사용자에게 *어떻게 할지*를 묻지 말고 *이렇게 하세요*를 디폴트로 제시한다. 유연성은 혼란을 만든다.

### 3.3 시각적 위계
작업 영역(데이터 테이블·결정 패널)은 **강조**하고, 사이드바·헤더는 **흐리게(recede)**. 색은 정보 전달 수단이지 장식이 아니다.

**중요한 위계 원칙 — 라벨은 데이터 absent보다 흐리면 안 된다.**
분류·그룹 라벨("카테고리", "성별", "기간" 같은 헤더 텍스트)이 그 안의 데이터 칩 중 *비활성/없음 상태*보다 흐리면 정보 위계가 거꾸로다. 사용자는 "이 칩이 뭐의 분류인지" 모르게 된다.

| 영역 | 색·웨이트 |
|---|---|
| 데이터 칩 (활성/present) | `text-primary` 또는 솔리드 배경 + 강조색, weight 600 |
| 분류 라벨 ("카테고리" 등) | `text-secondary`, weight 500 |
| 데이터 칩 (비활성/absent) | `text-tertiary`, opacity 0.7 |

라벨은 칩(데이터)보다 폰트 *크기*는 작거나 같지만, 비활성 칩보다는 *명확*해야 한다 — "데이터 < 라벨 < 활성 데이터"가 아니라 "비활성 데이터 ≤ 라벨 < 활성 데이터" 순서다. 사용자가 봐야 할 정보(칩 본체)와 그 *맥락*(라벨)이 둘 다 살아 있어야 화면이 읽힌다.

### 3.4 정보의 성격을 규정하는 "섹션 제목"

표·차트·카드 등의 정보 블록 *위*에는, 그 블록이 **무엇에 관한 정보인지** 한 줄로 알려주는 **섹션 제목**을 둔다. 이건 컬럼 라벨이나 메트릭 이름과는 다른 위계의 텍스트다.

**구분**:
- **섹션 제목** ("오퍼 핵심 요약 (전체)", "TOP 20 매입 추천", "등급 분포") — 정보의 *성격* 규정
- **컬럼 라벨 / 메트릭 이름** ("종합 등급", "오퍼 금액", "총 SKU") — 데이터의 *항목* 식별
- **데이터 값** ("€393K", "37.2%", "전반 패스 오퍼") — 실제 *내용*

위계 순서: **섹션 제목 > 데이터 값 (시각적 강조) > 컬럼 라벨 > 보조 텍스트**.

| 영역 | 토큰 |
|---|---|
| 섹션 제목 (스코프 — 메인) | `font-display`, `text-xl` (20px), weight 600, **color `text-primary` (화이트)** |
| 섹션 제목 (eyebrow — 보조) | `font-sans`, `text-xs` (11px), weight 500, color `text-tertiary`, uppercase + letter-spacing |
| 컬럼 라벨 | `font-sans`, `text-sm` (13px), weight 400, color `text-secondary` |
| 데이터 값 (메인 hero) | `font-display`, `text-3xl` (34px), weight 500, color `text-primary` |
| 데이터 값 (표 행) | `font-display`, `text-md` (15px), weight 500, color `text-primary` |

섹션 제목은 **반드시 화이트(`text-primary`)**로. 회색으로 흐리게 두면 "이 표가 뭐에 관한 건지"가 안 읽혀서 사용자가 데이터부터 해석해야 한다 — 정보 위계 거꾸로.

**위계 역전 — 스코프가 메인, eyebrow가 보조.**
한 화면에 *오퍼 핵심 요약 (전체) / 시즌별 / 브랜드별*처럼 같은 본문이 반복되는 섹션 제목을 여러 개 둘 때, 본문("오퍼 핵심 요약")은 매번 같으니 사용자가 진짜 알아야 할 *핵심*은 스코프(`전체`/`시즌별`/`브랜드별`)다. 이 경우 본문은 작은 회색 eyebrow로, 스코프를 큰 화이트 메인 제목으로 둔다.

```html
<h3>
  <span class="scope">전체</span>             ← 큰 화이트 (fs-xl 20px, weight 600)
  <span class="eyebrow">오퍼 핵심 요약</span>  ← 작은 회색 (fs-xs 11px, uppercase)
</h3>
```

같은 카드를 여러 스코프로 반복할 때 이 패턴을 쓰면 "이 카드가 어느 스코프인지"가 즉시 인지된다. 스코프가 단일하거나 (예: "TOP 20 매입 추천") 본문 자체가 다 다르면 메인 제목 한 줄로 충분하다.

### 3.4 마우스 우선, 키보드는 보너스
모든 작업은 마우스 클릭만으로 100% 가능. 단축키가 있으면 화면에 힌트 표시(예: 버튼 옆 `⌘K`).

### 3.5 결정 상태가 디자인보다 먼저
모든 항목은 명시적 상태를 갖는다 (`검토 전 → 검토 중 → 결정 완료 → 보류 → 제외`). 상태는 색 점(dot) 또는 배지(badge) 하나로 표시한다.

## 4. 시각 토큰

### 4.1 색상

```css
/* 라이트 모드 */
--background:      #FFFFFF;
--surface:         #FAFAFA;   /* 카드·패널 배경 */
--border:          #E5E5E5;
--text-primary:    #0A0A0A;
--text-secondary:  #737373;
--text-tertiary:   #A3A3A3;

/* 다크 모드 */
--background-dark:     #0A0A0A;
--surface-dark:        #171717;
--border-dark:         #262626;
--text-primary-dark:   #FAFAFA;
--text-secondary-dark: #A3A3A3;

/* 액션 컬러 — "지금 눌러야 할 단 하나"에만
 * Linear violet 채택 (참조 모델이 Linear이므로 시각도 Linear 톤으로 통일).
 * 이전 버전: Toss blue #3182F6 (라이팅만 토스, 시각은 Linear라는 본 문서 1번 원칙과
 * 모순이라 2026-04 정정). */
--primary:       #5E6AD2;   /* Linear violet (brand) */
--primary-hover: #7078E0;
--primary-active:#4D58BC;

/* 상태 컬러 — 데이터 의미 전달용 */
--success: #00C471;   /* 긍정 변화, 결정 완료 */
--danger:  #F04438;   /* 부정 변화, 제외 */
--warning: #F79009;   /* 보류, 주의 */
--info:    #6938EF;   /* 정보 알림 */
```

**적용 규칙**:
- 한 화면에 액션 컬러는 최대 1~2곳
- 회색조 위주, 색은 데이터 의미 전달에만
- 다크/라이트 모드 양쪽 모두 검증

### 4.2 타이포그래피

**폰트**: Pretendard (한국어 기본), JetBrains Mono (숫자·코드)

**중요한 한 줄**: 숫자는 항상 `font-variant-numeric: tabular-nums`. 정렬이 어긋나면 표를 못 읽는다.

| 토큰 | 크기 / 행간 | 용도 |
|---|---|---|
| `text-xs` | 12 / 16 | 라벨, 캡션 |
| `text-sm` | 13 / 18 | 표 본문, 버튼 |
| `text-base` | 14 / 20 | 일반 본문 |
| `text-lg` | 16 / 24 | 섹션 헤딩 |
| `text-xl` | 20 / 28 | 페이지 타이틀 |
| `text-2xl` | 28 / 36 | KPI 숫자 |
| `text-3xl` | 36 / 44 | 메인 KPI |

**굵기**: 헤딩 600~700, 본문 400, 보조 400 + 회색.

### 4.3 간격·둥근 모서리·그림자

```css
/* 간격 (px) */
spacing: 4, 8, 12, 16, 24, 32, 48

/* 둥근 모서리 */
--radius-base:  6px;   /* 버튼, 인풋 */
--radius-card:  8px;   /* 카드 */
--radius-modal: 12px;  /* 모달 */

/* 그림자 */
--shadow-sm: 0 1px 2px rgba(0,0,0,0.04);   /* 카드 */
--shadow-md: 0 4px 12px rgba(0,0,0,0.08);  /* 드롭다운 */
--shadow-lg: 0 12px 32px rgba(0,0,0,0.12); /* 모달 */
```

**금지**: 그라데이션 배경, 글래스모피즘, 화려한 그림자, 12px 이상 둥근 모서리(모달 제외).

## 5. 레이아웃 패턴

### 5.1 기본 레이아웃: 좌측 사이드바 + 본문

```
┌────────┬─────────────────────────────────┐
│ 사이드 │ 상단 헤더 (검색, 필터, 액션)    │
│ 바     ├─────────────────────────────────┤
│ (흐림) │  본문 (강조)                    │
│ 220px  │  좌측: 리스트 · 우측: 상세      │
└────────┴─────────────────────────────────┘
```

- 사이드바: 220px 고정, 본문보다 1~2단계 흐린 톤
- `min-width: 1280px` 데스크톱 가정
- 모바일·태블릿 대응 안 함

### 5.2 마스터-디테일 (필수 패턴)

탭으로 화면을 분리하지 않는다. **좌측 리스트 + 우측 상세**를 한 화면에 둔다.

- 리스트 행 클릭 → 우측 패널 즉시 갱신 (페이지 전환 X)
- 우측 패널 안에서 시뮬레이션 가능 (입력 변경 → 즉시 결과)
- 페이지 전환·로딩 스피너 최소화

### 5.3 데이터 테이블

- 행 높이: 36~40px
- 줄무늬(zebra) 사용 안 함, 호버 시 배경색 변화
- 정렬·필터는 헤더에 상시 노출
- 가상 스크롤 또는 무한 스크롤 (페이지네이션은 차선)
- 우측 끝에 결정 상태 컬럼 고정

## 6. 컴포넌트 가이드

### 6.1 컴포넌트 출처 우선순위

1. **shadcn/ui** — 기본 부품 (버튼, 인풋, 다이얼로그, 표 등)
2. **Tremor Raw** — 차트, KPI 카드
3. **Lucide** — 아이콘
4. **TanStack Table** — 복잡한 표
5. **Recharts** — Tremor에 없는 차트

새 컴포넌트를 만들기 전에 항상 위 라이브러리를 먼저 확인한다.

### 6.2 자주 쓰는 shadcn 블록

| 블록 | 어디에 쓰나 |
|---|---|
| `dashboard-01` | KPI + 차트 + 표 → 분석 화면 base |
| `sidebar-07` | 사이드바 + 본문 → 전체 레이아웃 base |
| `data-table` | 정렬·필터 표 → 리스트 화면 base |

### 6.3 상태 표시

- **점(dot)**: 8px 원, 색상으로만 상태 표시 (테이블 행 같은 밀집 영역)
- **배지(badge)**: 텍스트 + 색상, 좀 더 명확히 보여줄 때
- 둘 다 색상 의미는 통일: 초록=완료/긍정, 빨강=제외/부정, 주황=보류, 보라=정보

### 6.4 버튼

- **Primary** (파란색): 한 화면에 최대 1~2개. *지금 눌러야 할 그것*에만
- **Secondary** (회색): 보조 액션
- **Ghost** (테두리만): 사이드 액션
- **Danger** (빨강): 제외·삭제 같은 되돌릴 수 없는 액션

라벨은 행동 동사로: ❌ "확인" / ✅ "매입 결정", "제외하기"

## 7. UX 라이팅 (한국어)

이 영역은 토스 스타일을 따른다.

### 7.1 톤

- 명령조 금지, 친근하지만 가볍지 않게
- "~할게요", "~해 주세요", "~예요/이에요"
- 영어 남용 금지: "프린시플" → "원칙", "코어 밸류" → "핵심 가치"

### 7.2 좋은 예 / 나쁜 예

| 상황 | ❌ 나쁜 예 | ✅ 좋은 예 |
|---|---|---|
| 빈 상태 | 데이터가 없습니다 | 아직 검토할 항목이 없어요. 엑셀을 업로드해 보세요. |
| 에러 | 잘못된 입력입니다 | 매입가는 숫자만 입력할 수 있어요. |
| 확인 | 정말 삭제하시겠습니까? | 이 항목을 제외할게요. 되돌릴 수 있어요. |
| 성공 | 저장 완료 | 12개 항목을 저장했어요. |
| 로딩 | Loading... | 잠시만요, 데이터를 불러오고 있어요. |

### 7.3 숫자 표기

- 큰 숫자는 한국어 단위: `1,234,000,000원` → `12억 3,400만원`
- 변화량은 부호 + 색: `▲ 18.2%` (초록), `▼ 5.1%` (빨강)
- 비율은 소수점 1자리: `14.4%`
- 표 안의 숫자는 모두 우측 정렬, tabular nums

## 8. Don't List (절대 하지 말 것)

1. 그라데이션 배경, 글래스모피즘, 화려한 애니메이션
2. 이모지 남용 (상태는 색 점 또는 배지로)
3. 한 화면에 3개 이상의 액션 컬러
4. 12px 이상 둥근 모서리 (모달 제외)
5. 불필요한 탭/스텝 (좌우 분할로 합칠 수 있는지 먼저 검토)
6. 모달 안에 모달
7. "확인", "취소" 같은 모호한 라벨 (행동 동사로)
8. 숫자에 비례 폰트 (정렬 어긋남, 항상 tabular nums)
9. 중요한 정보를 회색 텍스트로 (회색은 보조 전용)
10. 페이지 전체 로딩 스피너 (스켈레톤 UI 또는 부분 갱신)
11. 단축키 없이 작동 안 하는 기능
12. 모바일·태블릿 대응 (데스크톱 전용)

---

## 9. 결정이 애매할 때

```
질문: 이 화면/컴포넌트를 어떻게 만들지?

1. shadcn 블록에 비슷한 게 있는가?     → 있으면 그것부터
2. 정보 밀도가 충분한가?                → 부족하면 더 압축
3. 처음 보는 사람도 5~10분 안에 핵심   → 아니면 위계 재검토
   작업 가능한가?
4. 마우스만으로 100% 작동하는가?       → 단축키는 그 다음
5. 다크 모드에서도 정상인가?            → 항상 양쪽 확인
6. 헤비유저 vs 라이트유저 충돌?         → 헤비유저 편
```

---

## 10. 우선순위 (의심될 때)

1. 올바른 결정 > 빠른 결정
2. 헤비유저(MD) 효율 > 라이트유저 친절함
3. 정보 밀도 > 여백
4. 일관성(프로젝트 간) > 개별 화면 완성도
5. 마우스로 100% 가능 > 키보드 단축키
6. 데스크톱 전용 > 반응형
7. 상태의 명확성 > 시각적 화려함

## 11. 프로젝트별 특화

이 문서는 *전체 공통*이다. 각 프로젝트에서 추가 필요한 것은 그 프로젝트 루트의 `PROJECT.md`에 작성한다.

**`PROJECT.md`에 들어갈 것**:
- 그 프로젝트만의 화면 목록과 *5초 안에 답할 질문*
- 그 프로젝트만 쓰는 특수 컴포넌트
- 데이터 구조 (테이블·필드)
- 그 프로젝트만의 단축키 매핑

**충돌 시**: `PROJECT.md`가 이긴다. 단, 이 문서의 8번 "Don't List"는 절대 깨지 않는다.
