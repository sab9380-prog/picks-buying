## 2. 핵심 화면 흐름 (현재)

**입력 영역과 진단·제안 영역을 시각적으로 명확히 분리** — 사용자가 "어디에 입력하고, 어디서 결과를 보는지" 헷갈리지 않도록.

```
[헤더: NC PICKS 로고 | 사이트 타이틀 | 우측: 오퍼업로드/단건입력 + 상태바]
       ↓
┌─ 입력 영역 ───────────────────────────────────────────────────┐
│ [입력 폼:                                                      │
│   ① 오퍼 파일 업로드                                            │
│   ② 매입 고려 사항 (단일 textarea, 자유양식)                    │
│   ③ 분석 시작 버튼]                                            │
└──────────────────────────────────────────────────────────────┘
       ↓ (제출 → 입력 폼 접힘 + 결과로 스크롤)
═══ 분석 결과 ═════════════════════════════════════════════════
[분석 결과 헤더 + 환율 주석: "EUR × 1,740 · 2026-01 기준 (가정값)"]
[필터 칩: 카테고리 1줄 / 성별+시즌 1줄, 매칭 highlight 만, 클릭 X]
[Hero — '전체' 카드: scope(큰 화이트) + eyebrow("오퍼 핵심 요약" 작은 회색)
  · 종합행 fs-3xl 34px, 한화는 별도 fs-xl 20px 화이트 라인]
[시즌별 카드 (별도): 같은 5컬럼 medium 양식, ht-num-medium fs-md 15px]
[브랜드별 카드 (별도): 같은 양식, qty 내림차순 정렬]
[진단 요약:
   - cons-box: 매입 고려 사항에서 추출 (매입율 / MOQ / 인코텀 / 리드타임)
   - verdict 카드 (권고 한 줄, 토스 톤, 톤별 색)
   - 내레이티브 단락 1개]
[탭: 전체 오퍼 / TOP 20 매입 추천 / 전체 SKU 표
   - 전체 오퍼 탭의 통계 8종을 4×2 그리드로 (등급분포 + 카테고리/성별/사이즈/컬러/시즌/수량분포/가격대)]
═══════════════════════════════════════════════════════════════
```

**원칙**: 결과 영역에는 *진단·분석*만 둔다. MD 컨텍스트(매입 고려 사항)는 입력 폼에서만 입력. 분석 후 메모를 추가하려면 헤더 "오퍼 업로드"를 다시 눌러 입력 폼을 펼친 뒤 재분석. 이는 **입력 vs 진단 영역의 시각적·기능적 분리**를 위한 의식적 선택이다.

## 2.1 매입 고려 사항 텍스트 → 분석 반영

자유양식 textarea에서 **룰 기반 정규식**으로 핵심 조건을 추출해 진단 영역에 표시한다 (`extractConsiderations` in `dashboard.js`).

| 추출 항목 | 패턴 예시 | 결과 |
|---|---|---|
| 매입율 / 할인율 | `80% off retail`, `Offer at 20% of RRP` | `discountPct: 80, buyRate: 0.20` |
| MOQ | `Minimum total order - 50k EUR`, `MOQ 100 USD` | `{ value: 50000, currency: 'EUR' }` |
| 인코텀 | `Ex-works Dubai`, `FOB Busan`, `DDP Seoul` | `{ term: 'EXW', location: 'Dubai' }` |
| 리드타임 | `Delivery 6 to 8 weeks`, `ship in 4 weeks` | `{ min: 6, max: 8, unit: 'weeks' }` |

**진단 영역 동작**:
- 추출 결과를 `cons-box`로 명시 표시 (verdict 카드 위)
- **컨텍스트 매입율(buyRate)이 추출되면 분석 자체에 자동 적용** — 모든 `offer.jsc`를 `rrp × buyRate`로 강제. 원본은 `offer.jscOriginal`로 보존(비교용). 적용 메타(`ctxApplied`, `originalAvgBuyRate`)는 IndexedDB 컨텍스트 키와 함께 저장 → restore 시에도 진단에 반영.
- 추출 박스의 매입율 비교 칩에 "분석 적용됨" 보라 칩 표시 (원본 데이터 매입율도 같이 보여줌).
- 원본 데이터 매입율과 컨텍스트 매입율 차이 ≥10%p면: `cons-box-warn` 노랑 보더 + verdict 톤 caution으로 강제 + narrative에 경고 단락 ("벤더·BI 데이터 양쪽 확인 후 의사결정"). 컨텍스트는 이미 분석에 적용된 상태에서의 안내.

**적용 결정 근거**:
- 사용자가 컨텍스트에 "X% off"를 명시하면 그건 *오퍼 가격 정의*이므로 분석 기준이 되어야 한다.
- 데이터 JSC 컬럼이 부재하거나(0%) 컨텍스트와 다른 의미일 때 컨텍스트가 더 정확한 정보다.
- 사용자가 잘못된 컨텍스트를 입력할 위험은 충돌 경고와 원본 매입율 표시로 완화.

**한계**:
- 100% 정확하지 않음. 변형 표현(예: "MOQ 100 units"처럼 통화 없는 경우)은 안 잡힘.
- 수치만 잡고 컨텍스트(예: "MOQ 100 units"의 단위)는 보존 안 됨.
- 컨텍스트 매입율 적용 시 SKU별 가격 차이가 평탄화됨 (모든 SKU가 동일 매입율) — JSC 부재 케이스에선 자연스럽지만 JSC 있는 케이스에선 SKU별 점수·등급 차이가 줄어듦.
- 후속 라운드 K로 "원본 데이터로 돌아가기" 토글 추가 검토.

## 3. parent DESIGN.md와의 차이점 (예외 처리)

### 3.1 의존성 정책 — `의존성 0` 유지
parent `DESIGN.md` 6.1은 shadcn/ui + Tremor + Recharts를 권장하지만 smart-buy는 **vanilla JS + ES Modules + 인라인 SVG 차트**를 유지한다. 이유:
- picks-v5.html과 한 페이지로 합쳐질 때 라이브러리 충돌 회피
- 단독 페이지 단순성 (의존성 트리 없음)
- 빌드 도구 없이도 정적 호스팅 가능

**예외 허용 라이브러리** (CDN으로만):
- Pretendard Variable (한글 폰트)
- Inter (영문 폰트)
- SheetJS (Excel 파싱, 동적 로드)

런타임 JS 라이브러리 추가는 0을 유지한다.

### 3.2 다크 모드 only
parent `DESIGN.md` 1번은 "다크/라이트 모드 양쪽 지원"을 명시하지만, smart-buy는 현재 **다크 모드 전용**이다 (1시간+ 사용 헤비유저 대상). 라이트 모드 토큰은 정의되지 않았다.
**후속 라운드 예정**: 라이트 토큰 추가 + 자동 전환.

### 3.3 레이아웃 — 사이드바 없음
parent `DESIGN.md` 5번은 "좌측 사이드바 + 본문" 기본 레이아웃을 권장하지만, smart-buy는 단독 페이지라 **헤더 + 단일 컬럼**이다. picks-v5 통합 후 사이드바 패턴을 따를 예정.

### 3.4 마스터-디테일 미적용
parent `DESIGN.md` 5.2는 마스터-디테일을 필수 패턴으로 두지만, smart-buy의 SKU 표는 **탭 + 디테일 패널** 패턴이다. 이는 v12 호환성 + 한 화면에 보이는 정보 우선이라는 트레이드오프.

### 3.5 색상 — Linear violet primary
parent `DESIGN.md` 4.1의 primary는 Linear violet (`#5E6AD2`)으로 통일됨 (2026-04). 토스 블루는 채택하지 않는다.

### 3.6 진단 텍스트 라이팅
verdict 카드 한 줄은 **토스 톤** ("~예요/이에요") 적용. 내레이티브 단락은 분석 보고서 톤 ("~이다")을 유지하되, 후속 라운드에서 톤 통일 검토.

## 4. 색상·토큰 — `design-tokens.css` 단일 출처

모든 시각 토큰은 `./design-tokens.css`에 정의된다. **JS·HTML에서 inline 색상·폰트·크기 금지**. CSS 클래스만 사용한다.

이 프로젝트만의 추가 토큰 (parent `DESIGN.md`에 없음):
| 토큰 | 값 | 용도 |
|---|---|---|
| `--fs-display` | 48px | hero 큰 숫자 (종합행) |
| `--fs-display-lg` | 64px | huge display (단일 큰 metric) |
| `--bg-elevated` | `#131318` | sticky header, panels |
| `--bg-card` | `#17171D` | card / dropdown / row hover |
| `--bg-input` | `#0F0F14` | sunken input fields |
| `--accent-subtle` | rgba(94,106,210,0.14) | accent muted bg |
| `--accent-muted` | rgba(94,106,210,0.08) | accent subtler bg |
| `--chart-color-1`~`10` | jewel tone | 차트 막대 다색 팔레트 |
| `--r-2xl` | 16px | hero card corner (parent의 12px 이상 금지 예외, 모달 외) |

**`--r-2xl` 예외**: hero card는 큰 숫자 강조 + 둥근 모서리로 시각적 경계 형성. parent `DESIGN.md` 4.3의 "12px 이상 금지"는 모달 외에 예외를 두지 않지만, smart-buy는 hero 1개에 한정해 사용. 다른 곳엔 안 쓴다.

## 6. FX_RATES — 가정값 (`dashboard.js` 안)

| 통화 | KRW × | 비고 |
|---|---|---|
| EUR | 1740 | v12 보존 (JSC 환산 기준) |
| USD | 1350 | 2025~26 평균 |
| GBP | 1780 | |
| JPY | 9.4 | |
| CNY | 190 | |
| CHF | 1500 | |

**기준일**: `FX_BASE_DATE = '2026-01'` (가정값). `fxAnnotation()` 헬퍼가 결과 영역 헤더에 `환율: EUR × 1,740 · 2026-01 기준 (가정값)` 주석으로 노출.

**한화 강조**: 종합행에서 외화 큰 숫자 바로 아래에 한화 환산이 별도 큰 라인(`ht-num-krw`, fs-xl 20px, 화이트, weight 500)으로 분리. RRP는 그 아래 작게. 시즌·브랜드 행의 한화 환산도 `ht-sub-sm`(12px, text-secondary, weight medium)으로 강조.

**중요**: 외환 변동 미반영. 진단 요약과 결과 헤더에 "가정값 · 외환변동 미반영"을 명시한다.

## 8. 후속 라운드 후보

이번 라운드에서 처리한 것 외 남은 작업:

- **A. 휴리스틱 보정** — 도매 깊은 할인 케이스 + RRP_PSYCH_FACTOR 차등
- **B. 매핑 미리보기 UI** — 자동탐지 결과 사용자 보정 + vendor-formats/*.json 자동 저장 (10% 미인식 보완)
- **C. 카테고리 마스터 매핑** — raw → 표준 10분류 + 원본 보존 + classify 정확도 측정
- **D. 환율 실시간/정기 갱신** — 현재 가정값 → API 또는 정기 update
- **E. picks-v5.html 통합 트리거** — 단독 페이지에서 통합 페이지로
- **F. 진단 항목 6개 추가** — 실 BI 데이터 후
- **G. 라이트 모드 토큰** — parent DESIGN.md 정합성
- **H. 표 가독성** — 3000+ SKU 정렬·검색·필터·가상 스크롤·결정상태 컬럼 우측 고정
- **I. 진단 narrative 토스 톤 통일** — 분석 보고서 톤 → 토스 톤
- **J. 색상·타이포 스케일 통일** — `--fs-xs` 11→12, signal 색상 등 parent DESIGN.md 정합
- **K. 결과 영역에서 "매입 고려 사항" 보기·수정** — 현재는 입력 폼에서만. 분석 후 메모 추가 시 재분석 필요. 작은 "수정" 버튼으로 입력 폼 다시 열기 검토.
- **L. 컨텍스트 매입율 토글** — 자동 적용 외에 "원본 데이터로 돌아가기" 토글. 사용자가 데이터 vs 컨텍스트 케이스별로 선택 가능.
- **M. 브랜드 카드 상위 N + 기타 합산** — 브랜드가 50+ 인 경우 카드 길어짐. 정보 밀도 보존하려면 상위 N + "기타 K개" 합산 행.
- **N. 통계 8종 등급 분포 시각 보정** — 다른 빈도 통계와 시각 패턴 다름. 4×2 그리드 안에서 어색하면 다시 분리 검토.
