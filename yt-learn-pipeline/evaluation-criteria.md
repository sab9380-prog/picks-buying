# YouTube 학습 파이프라인 — 평가 기준 v3

> 평가자 에이전트용 | 생성자 에이전트의 구현 결과를 검증할 때 참고하는 기준표

---

## 평가 원칙

1. **생성자와 평가자는 반드시 분리한다.** 코드를 만든 에이전트가 자기 코드를 평가하지 않는다.
2. **합격/불합격이 아닌 점수형 평가.** 각 항목을 10점 만점으로 채점한다.
3. **7점 미만 항목은 구체적 수정 지시와 함께 생성자에게 반환한다.**
4. **전 항목 7점 이상이면 해당 기능을 승인하고 다음 기능으로 넘어간다.**

---

## 평가 항목 (5개 영역, 각 10점)

### 1. 기능 정확성 (Functional Accuracy) — 10점

파이프라인이 product-plan.md의 기능 요구사항을 정확히 수행하는가?

| 점수 | 기준 |
|---|---|
| 9~10 | 모든 기능이 명세대로 동작하고, 엣지 케이스(중복 URL, 자막 없음, API 에러 등)도 처리됨 |
| 7~8 | 핵심 기능은 동작하나, 일부 엣지 케이스에서 오류 발생 |
| 5~6 | 주요 기능 중 1개 이상 동작하지 않음 |
| 3~4 | 자막 추출 또는 Claude 분석 단계에서 실패 |
| 1~2 | Slack 트리거부터 동작하지 않음 |

**필수 체크리스트:**
- [ ] Slack #yt-learn에 YouTube URL을 보내면 파이프라인이 자동 시작되는가?
- [ ] youtube.com/watch, youtu.be, /shorts/ 등 다양한 URL 형식에서 Video ID가 정확히 추출되는가?
- [ ] URL이 아닌 일반 메시지(텍스트, 이모지 등)는 무시되는가?
- [ ] 같은 URL을 두 번 보냈을 때 Notion에 중복 레코드가 생기지 않는가?
- [ ] Notion Videos DB에 레코드가 자동 생성되고, 4계층 필드가 모두 채워지는가?
- [ ] Tasks DB에 next_action 레코드가 자동 생성되고, Videos DB와 Relation이 연결되는가?
- [ ] 처리 완료 시 Slack에 "✅ 분석 완료" 알림이 돌아오는가?
- [ ] 처리 실패 시 Slack에 "❌ 실패" 알림이 사유와 함께 돌아오는가?

---

### 2. 에러 복원력 (Error Resilience) — 10점

예외 상황에서 파이프라인이 깨지지 않고 적절히 대응하는가?

| 점수 | 기준 |
|---|---|
| 9~10 | 모든 실패 시나리오에서 Status 표시 + Slack 알림 + 다음 레코드 처리 계속 |
| 7~8 | 주요 실패(자막 없음, API 에러)는 처리되나 드문 케이스는 미처리 |
| 5~6 | 실패 시 Status는 바뀌지만 Slack 알림이 안 옴 |
| 3~4 | 실패 시 플로우 전체가 멈춤 |
| 1~2 | 에러 핸들링이 없음 |

**필수 테스트 시나리오:**
- [ ] 자막이 없는 영상 URL → Status "No Transcript" + Slack 알림, 다음 레코드 계속 처리
- [ ] 잘못된 URL (youtube.com이 아닌 링크) → 무시 또는 Status "Error" + Slack 알림
- [ ] Claude API 429 (Rate Limit) → 재시도 후 정상 처리
- [ ] Notion API 실패 → Status "Error" + Slack 알림
- [ ] 영상 5개를 10초 간격으로 연속 공유 → 순차 처리, 누락 없음

### 3. 추출 품질 (Extraction Quality) — 10점

**이 항목이 이 파이프라인에서 가장 중요한 평가 영역이다.** Claude가 추출한 4계층 내용이 실제로 학습과 실행에 도움이 되는 수준인가?

| 점수 | 기준 |
|---|---|
| 9~10 | 아래 4개 세부 기준 모두 충족 |
| 7~8 | 계층 1~2는 우수하나, 계층 3~4에서 일부 추상적 |
| 5~6 | 계층 1은 괜찮으나, 계층 2~4가 일반적 조언 수준 |
| 3~4 | one_line_thesis조차 영상 핵심을 못 짚음 |
| 1~2 | JSON 파싱 실패 또는 빈 결과 |

**세부 기준 (4개):**

**① 계층 1 — 핵심을 짚었는가?**
- one_line_thesis가 영상의 핵심 주장을 정확히 담고 있는가? (영상 주제 설명이 아니라 "주장"이어야 함)
- framework이 영상에 실제로 등장한 사고 모델인가? (없는 걸 만들어내면 감점)

**② 계층 2 — 내 상황에 연결되었는가?**
- relevance_to_picks가 "유통업에 도움됨" 같은 뭉뚱그린 문장이 아니라, Picks의 구체적 업무(브랜드 매입, AI 전략, 역기획 등)에 맵핑되었는가?
- challenges_assumption이 실제로 기존 가정을 뒤집는 내용인가? (없는데 억지로 만들면 감점 — "없음"이 정직한 답)

**③ 계층 3 — 적용이 구체적인가?**
- apply_to_current가 "~를 검토해보기"가 아니라 "~에서 ~를 ~으로 바꾸기" 수준인가?
- 적용 대상이 실제 진행 중인 프로젝트(AI 전략 덱, 회의 자동화, 매입 프로세스 등)와 연결되는가?

**④ 계층 4 — 바로 실행할 수 있는가?**
- next_action이 한 문장으로 읽었을 때 "아, 이거 하면 되겠다"가 즉시 떠오르는가?
- deadline과 effort가 영상 내용의 긴급도에 비례하여 합리적인가?

**평가 방법:**
- 실제 영상 3개 이상으로 테스트 (OPR 영상 1개 + AI/테크 영상 1개 + 혼합 1개)
- 각 영상마다 "사람이 직접 정리했으면 이렇게 썼을까?"로 비교 판단

### 4. 운영 안정성 (Operational Stability) — 10점

연속 실행, 반복 실행에서 문제없이 돌아가는가?

| 점수 | 기준 |
|---|---|
| 9~10 | 연속 10개 영상 처리 시 전부 성공, Rate Limit 위반 없음 |
| 7~8 | 연속 5개는 성공, 10개부터 간헐적 429 에러 발생하나 재시도로 복구 |
| 5~6 | 3개 이상 연속 처리 시 Rate Limit으로 실패가 잦음 |
| 3~4 | 1개는 되지만 2개째부터 타이밍 문제 발생 |
| 1~2 | 단일 영상도 간헐적으로 실패 |

**필수 테스트:**
- [ ] 영상 1개 공유 → Slack 알림까지 전체 성공
- [ ] 영상 5개 연속 공유 → 전부 순차 처리 완료
- [ ] 같은 URL 재공유 → 중복 처리 없음 (멱등성)
- [ ] 24시간 후 새 영상 공유 → 정상 동작

---

### 5. 유지보수성 (Maintainability) — 10점

비개발자인 재열이 나중에 수정·확장할 수 있는 구조인가?

| 점수 | 기준 |
|---|---|
| 9~10 | 각 단계가 명확히 분리, 변수명 직관적, 주석 충분, 프롬프트 수정 포인트 명확 |
| 7~8 | 구조는 깔끔하나, 일부 복잡한 표현식에 설명 부족 |
| 5~6 | 동작하지만 어디를 고쳐야 할지 파악하기 어려움 |
| 3~4 | 하나의 거대한 플로우에 모든 것이 섞여 있음 |
| 1~2 | 이해할 수 없는 구조 |

**체크리스트:**
- [ ] Power Automate 플로우가 Step별로 Scope 또는 주석으로 구분되어 있는가?
- [ ] API 키, DB ID, Slack 채널명 등 설정값이 변수로 분리되어 있는가? (하드코딩 아닌지)
- [ ] Claude 시스템 프롬프트를 수정하고 싶을 때 어디를 바꿔야 하는지 바로 찾을 수 있는가?
- [ ] 4계층 추출 구조에 필드를 추가하고 싶을 때 수정 포인트가 3곳 이하인가? (프롬프트 + JSON 스키마 + Notion DB)
- [ ] 입력 채널을 추가하고 싶을 때 (예: 다른 Slack 채널, 이메일) 확장이 쉬운 구조인가?

- 필수 테스트 시나리오 전체 통과
- 연속 5개 영상 end-to-end 처리 성공 (Slack 공유 → Slack 완료 알림)
- **추출 품질 항목에서 반드시 8점 이상** (이 파이프라인의 핵심 가치)
